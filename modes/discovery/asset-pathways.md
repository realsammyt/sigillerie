# Asset Pathways

## Orientation

Phase 4 is the core of Discovery. Every other phase feeds it. Each asset type has three options. User picks one per slot. Mix-and-match across slots is supported (`A.logo + B.palette`). Some assets get generated by the agent. Some get acquired from the user, from stock, or from CC0 sources. Logos can be generated. Real product photos cannot. When acquisition fails, the slot records as gap in `discovery.json#gaps`. The agent never hallucinates a missing asset.

---

## Logo

Three routes. Run in parallel when API budget allows. Else pick one route, ship five variants from it.

**Route A, Recraft.ai.** Vector-native SVG output. Best for clean geometric marks and wordmarks. URL: https://www.recraft.ai. Prompt assembled from chosen direction's DNA + vibe-words + anti-brand list. Output: native SVG, no vectorization step.

**Route B, Brandmark or Looka.** Entrepreneur kit (logo + favicon + business card mock + social avatar). Brandmark: https://brandmark.io. Looka: https://looka.com. Best when user wants a starter package, not just the mark.

**Route C, Flux + logo-LoRA, or Midjourney v7.** Raster generation, then vectorize via potrace or vector.fm. Best for textured marks, hand-drawn feel, or unusual silhouettes. Vectorization always loses fidelity, so this route is for cases where the texture is the point.

**Bonus routes (when budget allows).** Claude SVG direct output. Gemini `gemini-3.1-flash-image-preview`.

**User picks one of three (A, B, C).** Or asks for `none, regenerate with [feedback]`.

**Writes:**
- `assets/<brand>-brand/logo-option-a.svg`
- `assets/<brand>-brand/logo-option-b.svg`
- `assets/<brand>-brand/logo-option-c.svg`
- chosen one copied to `assets/<brand>-brand/logo.svg`
- `discovery.json#assets.logo = { chosen, source, prompt, alternatives }`

## Palette

Three routes.

**Route A, Extract from moodboard.** k-means via chroma.js on a flagged moodboard image. Anchors palette to taste already confirmed in Phase 2.

**Route B, Generate from vibe-words.** Khroma (https://khroma.co) or Realtime Colors (https://realtimecolors.com). Algorithmic, fast, sometimes generic.

**Route C, Philosophy default.** Pull from the chosen direction's default palette in `direction-library.md`. E.g. Pentagram defaults to high-contrast neutrals plus one signal color. Hara defaults to warm whites and one ink.

**User picks one of three.** Or composes (`A but with B's accent`).

**Writes:**
- `assets/<brand>-brand/palette-option-{a,b,c}.css` (CSS custom properties)
- `assets/<brand>-brand/palette-option-{a,b,c}.json` (Style Dictionary tokens)
- chosen copied to `palette.css` + `palette.json`
- provenance logged (`source: "moodboard image #4"` or `"Khroma seed: warm,technical,calm"`)

## Typography

Three pairings. Each pairing is display + body + mono. All families filtered to free commercial license: Google Fonts, GitHub Fonts, SIL Open Font License.

**Sources:** Fontjoy (https://fontjoy.com) for pairings, Typewolf (https://www.typewolf.com) for editorial reference. Filter list against fonts.google.com and the SIL OFL registry.

**CJK fallback.** When `intake.locale === "zh"` (or any East Asian locale), each pairing carries a CJK companion: Source Han Sans, Noto Sans CJK, or LXGW Wenkai. Mono fallback: Sarasa Mono.

**Writes:**
- `assets/<brand>-brand/type-stack-option-{a,b,c}.css`
- comparison HTML at `choosers/phase-4-type.html`
- chosen copied to `type-stack.css`
- license noted per family in `discovery.json#assets.typography.license`

---

## Iconography

Pick one system. Not three. Icons in Discovery are family choice, not custom design. Custom icons belong in production.

**Three families to pick from:**
- Lucide (https://lucide.dev), default, 1500+ glyphs, MIT
- Phosphor (https://phosphoricons.com), six weights, MIT
- Tabler (https://tabler.io), 4000+ outline, MIT

**Defaults.** Pentagram-flavored direction → Lucide. Field.io-flavored direction → Phosphor (the weight axis suits motion work). Hara-flavored direction → Lucide thin or Phosphor light.

**Writes:**
- `assets/<brand>-brand/icon-system.txt` (one line: `Lucide@0.x` plus link)
- `discovery.json#assets.iconography.chosen`

## Product photography

Acquired, never generated when the product is real. This is the hard line.

**Three sources:**

**Source 1, User-provided files.** Phone camera, DSLR, prior brand archive. Agent ingests, normalizes (sRGB, max 4096px long edge), preserves originals.

**Source 2, 3D render.** Spline (https://spline.design), Blender, or pmndrs/assets material library. Use when the product is software, an abstract object, or genuinely 3D-modelable hardware. Output as PNG with alpha.

**Source 3, AI image gen with reference photo.** nano-banana-pro or Flux Kontext. Reference-conditioned only. Never text-to-image for a real product. The reference grounds the output in the actual thing.

**Gap path.** No files, no model, no reference photo → record `status: "gap", severity: "high", next_action: "user must shoot or commission"`. Producer mode flags this before delivery. Never hallucinate.

**Writes:**
- `assets/<brand>-brand/product-hero.png` and supporting shots
- `discovery.json#assets.product_imagery = { status, files, source }`

---

## UI screenshots

Two paths fork on whether the product exists.

**Real product exists.** Capture with Playwright. Three viewports (mobile, tablet, desktop). Three states (default, hover/focus, scrolled). Real screens, real copy.

**No product yet.** Three mockups in the chosen direction's style:
- Figma export (manual, when the user has a Figma file)
- v0.app (https://v0.app) or Google Stitch, generate from the brand-spec
- Hand-coded HTML rendered to PNG via Playwright

**Hard rule.** UI of a real product is never AI-generated as if it were a screenshot. Mock UI for a not-yet-built product is fine. The boundary is whether the user is implying the screenshot represents reality.

**Writes:**
- `assets/<brand>-brand/ui-home.png`, `ui-feature-{1,2,3}.png`
- `discovery.json#assets.ui_screenshots = { status: "captured" | "mocked" | "gap" }`

## Voice & tone

Pick a framework. Three options.

**Option A, Mailchimp voice/tone matrix.** Voice is fixed, tone shifts per context. Eight contexts × tone adjustments.
**Option B, Atlassian voice principles.** Four principles, applied to every surface.
**Option C, 3-axis spec.** formal↔casual, serious↔playful, dry↔warm. Each axis a 1–5 dial.

Agent fills the chosen framework from intake (vibe-words, audience, anti-brands). Returns a 1-page voice card with six example sentences across surfaces (button, error, onboarding, marketing headline, support reply, social post).

**Writes:**
- `assets/<brand>-brand/voice.md`
- `choosers/phase-4-voice.html` (3 cards)
- `discovery.json#assets.voice = { framework, card }`

---

## Audio brand

Opt-in. Fires when intake mentions sound, video, kiosk, or when `--audio` flag is set. Discovery emits an `## Audio Brand` block in `brand-spec.md` and hands off to the generative-audio capability.

**What Discovery does:** captures sonic vibe-words (warm, glitchy, sparse, anthemic), tempo target, instrumentation hints, and one or two reference tracks from user.

**What Discovery does not do:** generate the audio. That's `capabilities/generative-audio/brand-audio-spec.md`. Discovery hands off.

**Writes:**
- `## Audio Brand` block appended to `brand-spec.md`
- `discovery.json#assets.audio_brand = { handoff: true, ref_tracks, vibe_words }`

---

## 3D assets

Fires when `intake.first_surface ∈ {ar, xr, 3d-deck, model-viewer}`.

**HDRI environment.** Three options from Poly Haven (https://polyhaven.com). All CC0. Filtered by direction's lighting mood (Hara → soft overcast; Field.io → studio rim-light; Pentagram → neutral grey room).

**GLB / GLTF.** Three routes:
- Spline export (https://spline.design)
- Blender + AI texture (Substance, or Flux for albedo)
- Meshy or Luma Genie text-to-3D

**USDZ for Apple AR.** Convert via Reality Composer Pro CLI (`xcrun realityconvert`) or `usdzconvert` from Apple's USDZ tools. Native USDZ export from Blender 4.2+ when topology is simple.

**Material tokens.** PBR maps (base color, roughness, metallic, normal) tied to the chosen palette. Pulls from pmndrs/assets (https://github.com/pmndrs/assets) when a stock match exists.

**Writes:**
- `3d/hero.glb`, `3d/hero.usdz`, `3d/env.hdr`, `3d/materials.json`
- `## 3D Assets` block in `brand-spec.md`

## Generation vs acquisition boundary

| Asset | Generate? | Acquire? | Hallucinate when missing? |
|---|---|---|---|
| Logo | yes | sometimes (commission, license) | no |
| Palette | yes | yes (extract from moodboard) | no |
| Typography | pick from system | yes (free-license families) | no |
| Icon system | pick from system | yes (open-source family) | no |
| Product photo of real product | no | yes (user / commission / reference) | NEVER |
| UI of real product | no | yes (Playwright capture) | NEVER |
| Mock UI when no product yet | yes | n/a | n/a |
| Voice card | yes (from intake) | n/a | n/a |
| HDRI | n/a | yes (Poly Haven CC0) | no |
| GLB | yes (with reference / model) | yes (user-provided) | no |
| Audio brand | hands off to capability | n/a | no |

The hard rule: if the asset represents a real-world claim (this product, this UI, this person), it must be acquired. If the asset is a presentation choice (logo, palette, voice), it can be generated. The line is between claim and presentation.

## Copyright collision check

Before locking a logo, Phase 4 runs:

1. **TinEye reverse search** (https://tineye.com) on the candidate SVG rendered to PNG.
2. **Brandmark collision DB** lookup on the chosen wordmark / shapes.

**On hit:**
- Surface the source URL and similarity score in chat.
- Two branches: regenerate with constraint (`avoid X element`, `simpler silhouette`), or pivot to acquire (commission a designer, license stock from Creative Market, switch to wordmark only).
- Log `collision_event` in `discovery.json`:

```json
{ "ts": "...", "phase": 4, "type": "collision", "candidate": "logo-option-a.svg",
  "match_url": "...", "similarity": 0.87, "branch_taken": "regenerate" }
```

No silent overrides. If the user wants to ship despite the hit, that decision is logged too.

## Mix-and-match grammar

Choose-points accept:

- `A` / `B` / `C`, straight pick.
- `A.logo + B.palette`, compose across slots.
- `A but warmer`, single-axis adjustment, agent edits in place.
- `none, regenerate with [feedback]`, full re-roll with feedback in prompts.
- `B's typography on A's logo wordmark`, deeper compose, requires re-render of the affected option.

The grammar is consistent across phases. User learns it once.

---

## Anti-patterns

The signature failure: Recraft logo + Khroma palette + Inter type + Lucide icons. All four are tool defaults. The output looks like every 2024 startup. Generic-AI-brand fingerprint.

**Defense rule.** At least one asset must be anchored to moodboard or philosophy DNA, not pure tool output.

Acceptable anchoring:
- Palette extracted from a moodboard image (not Khroma-generated).
- Type pairing from Typewolf editorial reference (not Fontjoy default).
- Logo using a silhouette derived from a Phase 2 keep-image.
- Voice card built on a flinch (`not corporate, not Slack-cute`).

If all four assets came from defaults, agent flags it and asks the user to re-anchor at least one. No silent ship.

---

## Output writes

Every option is preserved. Discovery is non-destructive.

```
assets/<brand>-brand/
  logo-option-a.svg
  logo-option-b.svg
  logo-option-c.svg
  logo.svg                # symlink or copy of chosen
  palette-option-{a,b,c}.css
  palette-option-{a,b,c}.json
  palette.css             # chosen
  palette.json            # chosen
  type-stack-option-{a,b,c}.css
  type-stack.css          # chosen
  icon-system.txt
  product-hero.png        # acquired or rendered, never hallucinated
  ui-home.png             # captured or mocked
  voice.md
  audio/                  # only when audio brand opted in
3d/                       # only when first_surface is 3D
```

Re-runs are versioned, not destructive. Phase 4 re-run with new prompts adds `logo-option-d.svg` etc. The `events` log in `discovery.json` records every re-pick. Prior choices stay auditable. Forking produces `discovery.v2.json`.

The chosen path is always recoverable. The rejected paths are always recoverable. Discovery's job is to widen options before narrowing them, and to keep the trail.
